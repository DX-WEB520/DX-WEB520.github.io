# 客户端和服务器之间信息通信的基础知识

# 哪些场景会发生信息通信？

  • XMLHttpRequest [ajax|axios|$.ajax...|]/fetch 数据交互
  • 跨域处理:ajax、fetch、jsonp、postMessage
  • 资源获取 [html|css|js|image | 音视频...]
  • websocket
  • ...

请求 + 响应 = HTTP 事物

客户端和服务器之间传输的所有内容，统称为 HTTP 报文

我们可以通过谷歌浏览器的 NetWork 来查看所有客户端和服务器之前的通信信息。

# 从输入 URL 到看到页面，中间都经历了什么？

  1. URL 解析 [识别 URL]
  2. 检查缓存 [强缓存、协商缓存（针对于资源文件请求） & 本地存储（针对数据请求）]
  3. DNS 服务器解析 [域名解析：根据域名解析出服务器外网 IP]
  4. TCP 三次握手 [建立客户端和服务器见得的网络连接通道]
  5. 基于 HTTP/HTTPS 等传输协议，实现客户端和服务器之间的信息通信
  6. TCP 四次挥手 [把建立好的网络通道释放掉]
  7. 客户端渲染 [呈现出页面和效果]

# URL 解析

URI：统一资源标识符

  • URL：统一资源定位符
  • URN：统一资源名称

http://user:pass@www.baidu.com:80/st/index.html?xxx=xxx&xxx=xxx#video

# 传输协议

http(80)/https(443)/ftp(21)

# 用户名密码

use:pass (一般是不用的)

# 域名

www.baidu.com

  • 顶级域名：baidu.com
  • 一级域名：www.baidu.com
  • 二级域名：video.baidu.com
  • 三级域名：student.video.baidu.com

购买顶级域名自己后期可以分配二级、三级域名

域名的目的就是给对应的服务器外网 IP 起一个别名，好记，DNS 服务器是公共的。

www.baidu.com VS www.qq.com 跨域

www.baidu.com VS video.baidu.com 跨域 [主域相同，但是子域不同]

# 端口号

端口号就是用来区分一台服务器上的多个项目 [每一个项目其实都是一个服务]，取值范围 0-65535

# 请求资源路径名

st/index.html，基于路径找到客户端需要的资源文件。

我们看到的 URL 地址可能是重写后的 [看到的地址在文件目录汇总不存在] 比如一些动态地址不利于 SEO 优化，所以会进行重写变成.html 地址，利于 SEO 优化。

# 问号参数信息

xxx=xxx&xxx=xxx (x-www-form-urlencoded 格式)

把信息传递给服务器 GET 系列请求一般都是这样传递参数

如果是页面跳转，把信息传递给另一个页面

# HASH（哈希值）：#video

锚点定位

HASH 路由

# URL 编译问题

encodeURI & decodeURI : 编译空格和中文，一般编译整个 URL 中的信息

encodeURIComponent & decodeURIComponent：编译空格和中文以及一些特殊符号，所以一般只是用来编译传递的信息值的，而不是整个 URL [防止 URL 解析不了或者传递信息的乱码等问题]

--- 前后端都支持的 API

escape & unescape

--- 用于客户端页面信息传递或者一些信息的编译 [cookie 中的中文内容编译]

# 检查缓存

强缓存和协商缓存都是服务器端设置的，客户端浏览器自己会根据返回的一些信息，进行相关处理，无需前端单独设置什么。

缓存处理是基于 HTTP 网络层进行优化的一个非常重要的手段（针对资源文件请求）

# 强缓存：Expires/Cahche-Control

问题：本地缓存了文件，但是服务对应的资源文件更新了，我们如何保证获取的是最新的内容？

  • 请求资源文件的时候设置时间戳

    第一次，如果服务器资源有更新，再次发请求，保证时间戳不一样，这样就不会走本地的强缓存了，而是从新拉取最新的资源，.

  • 文件 HASH 名

    第一次，如果服务器资源更新，文件名字重新 HASH (webpack)

所以 HTML 文件不会做强缓存

# 协商缓存：Last-Modified/ETag

Last-Modified：记录服务器资源文件最后一次更新的时间

ETag：只要服务器资源文件改变，会生成一个不同的标识

当强缓存失效（或者不存在）[例如：html 可以做协商缓存]，会校验协商缓存 [前提是设置了协商缓存]

  • 每一次都会向服务器校验资源是否更新

# 数据缓存

需求：没有缓存的数据，我们从服务器拉取最新数据；有缓存数据，我们直接读取缓存的数据 [减少和服务器之间的交互频率，降低服务器压力，也可以提高页面的渲染速度...]

  • 页面不刷新，我们某些内容频繁操作，但是数据不是需要实时获取最新的，则可以做一下缓存；只要页面刷新，重新从服务器获取数据
  • 页面只要不关闭，我们读取缓存数据（针对不经常更新的数据）
  • 页面即使关闭，重新打开，我们也可以读取缓存中的数据（数据更新的频率更低，我们自己设定过期周期）

# 客户端实现数据存储的方案

  • 全局变量存储（vuex/redux）：页面刷新或者关闭后重新打开，之前存储的数据都没有了（内存释放问题导致的）
  • cookie
  • webStorage:localStorage / sessionStorage

# DNS

多服务器部署

  • 弊端：增加了 DNS 解析次数
  • 优势：资源合理利用，服务器抗压能力增强，提高 HTTP 并发性。
    • web 服务器（html/css/js 静态资源）
    • 图片服务器（一般图片的加载慢一点）
    • 数据服务器

对一小型项目来说，我们可以把项目部署在一个服务器上，如果是中大型项目，我们需要部署在多个服务器上，一般会做 DNS 预解析，让 GUI 渲染和 DNS 解析同时进行，使用
